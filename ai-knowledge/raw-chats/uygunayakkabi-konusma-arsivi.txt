Claude Prompt — Automation-Ready Admin Panel (Telegram-first, Admin override)
Reminder: this is an existing Next.js + Tailwind project. Do not start from scratch.

You are a senior Next.js + React + Tailwind engineer.

IMPORTANT:
Do NOT start from scratch.
This project already exists and is working.
Your job is to incrementally add an Admin Panel that is compatible with a future automation pipeline.

PROJECT:
Uygunayakkabi.com (Next.js + React + Tailwind)

BUSINESS MODEL:
No online payment checkout. Lead capture only (name + phone). Do not add payment features.

FUTURE PIPELINE (MUST BE COMPATIBLE):
Telegram (product photos + basic info) → Automation (n8n + AI) → Product Database → (Admin Panel + Website) + optional Instagram autopost.
Admin panel must be an "override/control layer" on top of the same Product Database used by automation.

TASK:
Implement an Admin Panel with authentication + RBAC + product management, with a data model designed to support Telegram-first ingestion.

1) Admin Authentication
- Admin panel protected by ID + PASSWORD login.
- Secure sessions (httpOnly cookies / server sessions).
- Password hashing (bcrypt or equivalent).
- Brute-force basic protection (rate limit or simple lockout after N attempts).

2) Users & RBAC (superadmin + sub accounts)
Roles:
- SUPERADMIN: full access, can manage users/roles.
- PRODUCT_MANAGER: can edit product fields (price, discount, title, category, images, active).
- INVENTORY_MANAGER: can edit stock only.
- VIEWER: read-only.

Enforce permissions BOTH in UI and API routes.

3) Product Management Scope (Override Layer)
Admin must be able to:
- edit price
- edit discount
- edit stock quantity
- toggle active/inactive
- manage images (but keep Telegram-first compatible; see images section below)
- edit category
- optionally edit title/description if present

Critical: Admin changes should not "break" future automation updates.
Implement fields to support source + overrides:
- source: "telegram" | "admin"
- createdBy (userId if admin-created)
- updatedBy
- lastSyncedAt (for future automation sync)
- lockFields (optional): a list of fields admin locks so automation won't overwrite them later (e.g. ["price","images"]).

If you think lockFields is too much for Phase-1, implement at least:
- source
- lastSyncedAt
- updatedBy

4) Images Strategy (Vercel-friendly, Telegram-first)
Do NOT implement local filesystem uploads as final storage.

Phase-1 requirement:
- Admin can manage images as a list of URLs:
  - Add/remove/reorder image URLs
  - Validate URLs
- Product model should support:
  images: string[] (array of URLs)
  primaryImage: string (optional)
Future-ready:
- Keep an "imageSource" concept (e.g. "telegram") so later we can switch to Cloudinary/S3 without refactoring.

5) Data Persistence (Shared with Automation)
You must inspect the repo and determine router type (App Router vs Pages Router) and current data source.
Implement a persistent DB that both Admin API and future automation can use.

Preferred:
- Prisma + SQLite for Phase-1 (simple, works locally; later can move to Postgres).
If Prisma integration is too heavy, propose a minimal alternative, but default to Prisma.

Create schema models:
- User (admin accounts)
- Role (or role enum)
- Product

Include migrations / schema setup steps.

6) Admin Routes / UI
Routes:
- /admin/login
- /admin (dashboard)
- /admin/products (list + search + edit)
- /admin/products/new (create)
- /admin/users (superadmin only)
- /admin/users/new (superadmin only)

UI features:
- Sidebar navigation + top bar + logout
- Products table:
  - thumbnail
  - name
  - category
  - price
  - discount
  - stock
  - active
  - source (telegram/admin)
  - updatedAt
- Product edit form:
  - price, discount, stock, active, category
  - images (multi URL manager)
  - save + validation
- Role-based UI:
  - hide user management for non-superadmin
  - disable product fields based on role (inventory manager stock-only)

7) API / Server Actions (Authorization required)
Implement product CRUD endpoints with RBAC checks.
Ensure any mutation checks the logged-in user role.

8) Seed / Bootstrap
- Create an initial SUPERADMIN user via a seed script or environment variables.
- Document how to run it locally.

9) Output Format (must be copy-paste usable)
First report:
- Which router (App/Pages) and which existing product page you found.
Then provide:
- FILE: path/to/file
  (full updated content)
for every added/changed file.

10) Constraints
- Minimal disruption: do not refactor unrelated pages.
- No payment, no checkout.
- Keep Tailwind styling consistent.
- Avoid adding many new libraries beyond auth + prisma.

Deliverables:
- working admin login
- RBAC enforced
- product CRUD with images URL management
- user management (superadmin)
- DB persistence compatible with future Telegram automation
